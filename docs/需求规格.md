## 1. 引言

### 1.1 目的

本文档定义「AI 内容获客系统」的功能需求、非功能需求、接口约束与验收标准，供产品、开发、测试使用。

### 1.2 产品愿景

为中小财务服务公司提供 **AI 辅助营销内容生产 → 内容管理 → 公众号发布（Mock/真实）→ 其他平台导出 → 内置 CRM 获客与跟进** 的一体化 SaaS 能力。支持 **一人加入多家公司、企业内自定义角色与功能权限、平台默认 AI 额度 + 租户自有 Key、销售组织与数据范围控制、可配置表单字段**，降低获客内容的人力成本，形成 **营销创作 → 营销活动 → 线索 → 客户 → 任务/跟进** 闭环，并预留多行业与知识库扩展。

### 1.3 术语

| 术语 | 定义 |
|------|------|
| 租户 / 公司（Tenant） | 一家企业，数据隔离单元 |
| 账号（User） | 全局登录身份（手机号 + 密码），与具体公司解耦 |
| 成员关系（Membership） | 某账号在某公司的成员身份，含角色与启用状态 |
| 企业角色（Tenant Role） | 公司内角色（内置 admin/editor 或自定义），绑定功能权限 |
| 当前公司（Active Tenant） | 用户登录后选定的工作空间，决定可见数据与权限 |
| 功能权限（Permission） | 可勾选的原子能力，如 `content.create`、`dashboard.view_all` |
| 行业包（Industry Pack） | 平台预置的行业模板、公共知识、合规规则集合 |
| 内容（Content） | 一次 AI 生成或编辑后的营销稿件 |
| Mock 发布 | 开发期模拟微信公众号发布，不调用微信 API |
| RAG | 检索增强生成，从知识库取片段注入 Prompt |
| LLM Provider | 大模型供应商标识，如 deepseek、openai_compatible、dashscope |
| platform_admin | 平台运营账号，管理全站，不参与普通企业权限体系 |
| 测试验证码 | 开发/测试环境短信 Mock 固定码 **1111**（`SMS_MOCK_CODE`） |
| 线索（Lead） | 尚未转化为客户的潜在客户记录 |
| 客户（Customer） | 已确认合作或意向明确的客户主体 |
| 联系人（Contact） | 隶属于客户的联系人；**权限随客户继承** |
| 跟进记录（Activity） | 对线索/客户的电话、拜访、备注等时间线记录 |
| 销售地区（Territory） | 租户内树形销售区域，用于数据范围与负责人归属 |
| 汇报关系（Sales Hierarchy） | 成员之间的上下级关系，用于 `list_team` 可见性 |
| 营销活动（Campaign） | 有周期与目标的营销获客容器，可关联内容与线索 |
| 任务（Task） | 销售待办（回访、报价跟进等），可关联线索/客户/活动 |
| 实体类型（entity_type） | 可配置字段的表单类型，如 `lead`、`customer`、`task` |
| 自定义字段（Custom Field） | 企业管理员在系统字段之外新增的字段，值存 `extra_data` |
| CRM Scope | 线索/客户/任务/活动等单据的数据可见性规则（见 §3.14.2） |

**UI 参考：** 企知道「科创空间」Web 与 APP/H5 的信息架构与视觉气质（见 §3.11）；CRM 列表交互参照现有 `ContentLibrary.vue` 模式。

**客户端范围（强制）：** 凡本文 **v0.3 起新增或修订** 的功能（含企业与权限、忘记密码、平台额度、选/切换公司等），均须 **Web 与 H5 同步实现、同步验收**，除非表中单独标注「仅 Web」。

### 1.4 范围

**当前版本包含：**

- Web 管理端（Vue 3）、移动端 uni-app（H5）
- 后端 API（FastAPI）、数据库（PostgreSQL / 开发期 SQLite）
- 可插拔 LLM；**平台默认 Key + 租户免费额度**（正文成功扣 1 次，方案不扣）
- 租户/平台知识库、品牌与个人偏好
- 企业 **角色与成员管理**、**一人多公司切换**
- 平台 **管理后台**（`/admin`，仅 Web）：租户/账号运营、转移管理员等
- **内置 SaaS CRM**（v0.5）：线索、客户、联系人、跟进、销售组织、任务、营销活动、可配置字段、**保存列表视图**、**数据导入**（见 §3.14～3.17）

**暂不包含：**

- 小红书/抖音官方自动发布 API
- 计费与支付
- **外部 Frappe CRM 对接**（已改为内置 CRM，见 §3.14）
- CRM 商机/看板/SLA、Twilio 通话（CRM-2 之后）
- 向量 RAG（P2，MVP 关键词检索）
- 微信 wx.login（P2）
- **微信小程序原生包**（当前 H5 为准；与 Web 能力对齐）

## 2. 用户与角色

### 2.1 用户画像

**小财务公司市场/运营人员**：非技术背景，需模板化创作与团队协作；内容涉及财税合规。

### 2.2 账号与公司

- 一个 **账号** 可拥有 **0～N 条成员关系**（加入多家公司）
- 每条成员关系：**一个公司 + 一个企业角色 + 启用/禁用**
- 登录后若多家公司，**必须先选择当前公司**；使用期间可 **切换公司**
- 切换公司后：数据、菜单、权限随 **当前公司** 变化；**个人偏好全局共用**（见 FR-BRAND-10）

### 2.3 企业内置角色

| 角色 code | 名称 | 说明 |
|-----------|------|------|
| `admin` | 企业管理员 | 系统内置，不可删除；**权限全开且不可削弱**；至少保留 1 名 admin |
| `editor` | 编辑 | 系统内置，不可删除；**新加入成员默认角色**；偏内容创作，默认 **无 CRM**（见 §2.8） |
| `sales` | 销售 | 系统内置，不可删除；一线销售：线索/客户/任务 **本人 scope**，CRM-1a 起预置 |
| `sales_manager` | 销售经理 | 系统内置，不可删除；**下级 + 地区** scope、分配负责人、公开视图 |
| `marketing` | 市场运营 | 系统内置，不可删除；营销活动 + 内容创作，轻量 CRM（活动线索） |

> **国内 CRM 对照：** 销售易「销售代表 / 销售经理 / 市场人员」角色划分；本系统用 **内置角色默认权限包** + 自定义角色微调。

### 2.4 自定义角色与内置角色管理

- 企业 admin 或拥有 `team.role.manage` 的角色可 **新建/编辑/删除** **自定义角色**
- **系统内置角色**（`admin` / `editor` / `sales` / `sales_manager` / `marketing`）**不可删除、不可改 code**
- `admin` 角色权限集 **固定为全部权限**，不可在 UI 中取消勾选
- 其余内置角色（`editor`、`sales`、`sales_manager`、`marketing`）：企业 **admin** 可 **调整权限勾选**（与自定义角色相同编辑器），但不可删除角色本身
- 通过 **功能权限勾选** 配置能力（见 §2.5、§2.8）

**CRM 列表范围规则（FR-CRM-SCOPE）：**

- 有 `*.list_all` → 当前租户全量
- 否则可见集合为以下 **并集**（满足任一即可）：
  1. `owner_user_id == 当前用户`（`list_own` 隐含）
  2. 负责人属于当前用户的 **汇报下级**（`list_team`）
  3. 单据 `territory_id` 落在用户 **可访问地区子树**内（`list_territory`；**地区负责人默认拥有该地区及子树**）
- API **服务端过滤**；前端隐藏仅辅助
- **联系人**不单独做 scope；随所属客户
- **任务** scope 与关联对象一致：按 `assignee_user_id` / `owner_user_id` / `territory_id` 套用同一并集规则

**企业 admin** 默认拥有全部 CRM 与 `crm.org.manage`、`crm.schema.manage`。

**忘记密码流程（FR-AUTH-18～21）：**

```
登录页 → 忘记密码
  → 输入手机号 → 获取验证码
  → 输入验证码 + 新密码 + 确认密码
  → 提交成功 → 返回登录页用新密码登录
  → （若多公司）再按 FR-AUTH-13 选择公司
```

**接口概要（公开，无需登录）：**

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/auth/password/forgot/send-code` | 发送找回密码验证码（body: phone） |
| POST | `/api/v1/auth/password/forgot/reset` | 验证码 + 新密码重置（body: phone, code, password） |

**与现有能力关系：**

| 能力 | 场景 | 是否登录 |
|------|------|----------|
| 密码登录 FR-AUTH-02 | 日常登录 | 否 → 是（JWT） |

**安全与体验（NFR，见 §5）：**

- 未注册手机号：发送验证码返回明确提示（不泄露是否存在，或统一文案「若已注册将发送验证码」— 实现时二选一并在测试说明）  
- 验证码有效期、发送间隔与登录验证码 **可共用配置**（`SMS_CODE_EXPIRE_SEC`、`SMS_SEND_INTERVAL_SEC`）  
- 建议验证码 **场景隔离**（login / reset_password），避免登录验证码被用于改密  
- 重置成功后 **使该账号既有 JWT 失效**（可选 P1：改密后清 token）  
- **测试环境：** `SMS_PROVIDER=mock`，`SMS_MOCK_CODE=1111`；登录与找回密码均使用该码；Web/H5 测试文档统一写明 **1111**

### 3.16 保存筛选的列表视图（FR-VIEW，v0.5.1）

> 在 §3.15 默认列偏好之上，支持将 **筛选条件、排序、列配置、关键字** 保存为可复用视图。参考 Frappe `CRM View Settings` 的 list 能力；**不做** kanban / group_by。

#### 3.16.1 数据模型 `entity_list_views`

| 字段 | 说明 |
|------|------|
| `tenant_id` | 租户 |
| `entity_type` | lead / customer / task / campaign |
| `name` | 视图名称 |
| `owner_user_id` | 创建人 |
| `is_public` | 租户内共享 |
| `is_pinned` | 钉在侧栏或视图切换器 |
| `is_default` | 该用户进入列表页默认视图（**每人每实体最多 1 个**） |
| `filters` | JSON，条件组 |
| `sort` | JSON，如 `[{ "field_key": "created_at", "dir": "desc" }]` |
| `columns` | JSON，同 §3.15；**为空**时回退默认列偏好 |
| `search_q` | 可选，保存的关键字搜索 |

**`filters` JSON 示例：**

```json
{
  "logic": "and",
  "conditions": [
    { "field_key": "status", "op": "in", "value": ["跟进中", "新建"] },
    { "field_key": "territory_id", "op": "eq", "value": "uuid-..." },
    { "field_key": "cf_annual_revenue", "op": "gte", "value": 100000 }
  ]
}
```

**筛选操作符（MVP）：**

| field_type | 操作符 |
|------------|--------|
| text / phone / email | `eq`, `neq`, `contains`, `is_empty` |
| select | `eq`, `neq`, `in` |
| multiselect | `contains_any`, `contains_all` |
| number / currency | `eq`, `gt`, `gte`, `lt`, `lte`, `between` |
| date / datetime | `eq`, `before`, `after`, `between` |
| user_ref / territory_ref | `eq`, `in` |
| checkbox | `eq` |

自定义字段筛选仅允许 `is_active=true` 的字段；PostgreSQL 走 `extra_data` JSON 路径，SQLite 开发期可应用层过滤。

#### 3.16.2 功能需求

| ID | 需求 | 优先级 | 验收标准 |
|----|------|--------|----------|
| FR-VIEW-01 | 多视图 | P1 | 每用户每实体可建多个私有视图 |

| FR-VIEW-05 | 钉选视图 | P1 | 钉选后出现在 Web 侧栏对应实体子入口 |
| FR-VIEW-06 | 默认视图 | P1 | 进入 `/leads` 等自动应用用户 `is_default` 视图 |

## 4. 数据与业务规则

### 4.1 内容状态机（v0.3 简化）

```
draft → scheduled → publishing → published
draft → publishing → published（具备 publish 权限）
publishing → failed → draft 或 retry
draft → exported（导出路径，按权限）
```

**BR-01：** AI 生成默认 draft，不自动发布。

**BR-02（修订）：** 不强制审核状态机；发布权限由 `content.publish` 控制。

**BR-03：** 所有业务查询使用 **active_tenant_id** 过滤；切换公司后隔离生效。

**BR-04～07：** 同 v0.2（industry_code、合规免责声明等）。

### 4.2 线索状态（CRM）

```
待跟进 → 跟进中 → 有意向 → 已转化（终态）
待跟进/跟进中/有意向 → 无意向 / 无效（终态）
```

**BR-CRM-01：** 线索默认 `owner_user_id` = 创建人。

**BR-CRM-02：** 客户 `converted_from_lead_id` 唯一；一条线索最多转化一次。

**BR-CRM-03：** 联系人必须挂 `customer_id`；删除客户时联系人级联软删或禁止删（实现二选一，须文档化）。

**BR-CRM-04：** 任务 `done` 后仍可见，不可从他人 scope 外访问。

**BR-CRM-05：** 营销活动 `ended` 后不可新增关联内容，可查看统计。

**BR-CRM-06：** CRM 与内容 **权限体系独立**；活动关联不改变内容 author 权限。

**BR-CRM-07～08：** 见 §3.14.12（种子字段 label、导入模板中文表头）。

19. A 创建任务指派自己 → 工作台「今日待办」+1；完成后状态 done
